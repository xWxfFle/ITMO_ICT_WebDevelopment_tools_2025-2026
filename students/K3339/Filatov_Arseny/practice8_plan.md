# Практическая работа 8. Планирование работ по проекту

**Проект:** веб-приложение для визуализации и анализа данных (BI-платформа) с поддержкой SQL-совместимых БД и генерацией SQL через ИИ.

---

## 1. Анализ ТЗ и план работ по проекту

| № этапа | Название этапа | Содержание работ | Сроки выполнения* | Необходимые ресурсы |
|:-------:|----------------|------------------|-------------------|---------------------|
| 1 | Исследование и уточнение требований | Интервью с пользователями, DBA и специалистом по данным; фиксация отчётности и ограничений по нагрузке и безопасности | 2–3 недели | ПМ/аналитик, представители бизнеса, DBA (консультации) |
| 2 | Архитектура и безопасность | Архитектура (UI, API, слой запросов к БД, ИИ); модель ролей; аудит и ограничения доступа к данным | 3–4 недели | Архитектор, консультация по ИБ |
| 3 | Инфраструктура и CI/CD | Репозитории, сборка/тесты, окружения dev/stage/prod, секреты, базовый мониторинг | 2–3 недели | DevOps, облако или VPS, CI |
| 4 | Подключение к БД | PostgreSQL, MySQL, ClickHouse, SQL Server: драйверы, пул соединений, таймауты, лимиты, политики read-only | 6–8 недель | 2 backend-разработчика, тестовые БД, ревью DBA |
| 5 | Ядро выполнения SQL и метаданные | Безопасное выполнение запросов; описание схемы для ИИ (`information_schema` и словарь бизнес-терминов) | 4–6 недель | Backend, специалист по данным |
| 6 | Визуализация и дашборды | Виджеты (таблица, графики, карты), drag-n-drop, сохранение и доступ по ролям | 8–12 недель | 2 frontend, при необходимости UX/UI |
| 7 | ИИ: NL → SQL → визуализация | Промпты RU/EN; валидация SQL; выполнение; автоматический выбор типа графика | 6–10 недель | Backend + данные/LLM, API ключи |
| 8 | Роли и администрирование | Учётные записи, роли (админ, аналитик, просмотр), журнал административных действий | 3–5 недель | Backend, frontend админки |
| 9 | Тестирование и приёмка | Функциональное и регрессионное тестирование; безопасность (SQLi, права); нагрузочные пробы | 4–6 недель | 1–2 QA, автотесты по мере готовности |
| 10 | Документация и ввод в эксплуатацию | Руководства, обучение пользователей, план релиза и отката | 2–3 недели | ПМ, техписание/разработчики |

\*Сроки условные; этапы **4–8** частично **параллелятся** после утверждения архитектуры. Полный объём ТЗ по календарю часто **~10–14 месяцев** при команде среднего размера.

## 2. Объём финальной лабораторной и список задач

**Что берём в финальную ЛР (узкий MVP):** одна СУБД (например PostgreSQL), один дашборд (таблица + один тип графика), упрощённые роли (аналитик / просмотр), демонстрация вопроса на RU/EN → сгенерированный SQL → результат **без** карт и **без** всех четырёх СУБД сразу.

| № | Задача | Информация для выполнения |
|---|--------|---------------------------|
| 1 | Зафиксировать пользовательские сценарии ЛР | Документ: подключение → вопрос на естественном языке → SQL → таблица/график. Пример: *«Покажи продажи по категориям за последний квартал»* при таблицах `sales`, `categories`. |
| 2 | Спроектировать подключение к БД | Только безопасный режим (например только `SELECT`), таймаут и лимит строк; пример конфигурации в `.env`; тестовый дамп 2–3 таблиц. |
| 3 | API выполнения SQL | REST: запрос `{ "sql": "..." }`, ответ `{ "columns": [...], "rows": [...] }`; явно описать запреты (DDL/DML или белый список). |
| 4 | Метаданные для промпта ИИ | Выгрузка схемы (таблицы, колонки, типы); пример фрагмента, уходящего в промпт. |
| 5 | Интеграция LLM | Промпт на русском и английском; формат ответа (только SQL). Golden-примеры: 3–5 вопросов и эталонный SQL. |
| 6 | Валидация сгенерированного SQL | Проверка на read-only, ограничение сложности; пример отклонённого запроса и сообщения пользователю. |
| 7 | Frontend: подключение к БД | Поля host, port, database, user, password; пароль не хранить в открытом виде на клиенте — только через backend/сессию. |
| 8 | Frontend: таблица и график | Библиотека графиков (Chart.js, ECharts и т.п.); тестовые данные вида `{ "category": "...", "amount": 123 }`. |
| 9 | Упрощённые роли | JWT или сессия с полем `role`: аналитик меняет запросы и сохраняет дашборд; viewer только читает готовое. Таблица соответствия ролей и маршрутов API. |
| 10 | README и ограничения ЛР | Запуск БД и приложения; скриншоты; список того, что **не** входит в ЛР (остальные СУБД, карты, полная админка пользователей). |

### 4.2. Страховые взносы и налоги с ФОТ (при найме по ТК)

У работодателя страховые взносы с выплат сотрудникам в типичной модели **не ниже ~25–30%** от начислений в фонд (конкретная ставка зависит от лимитов базы и льгот; для сметы закладывают **~28%** от суммы «гросс» ЗП по найму).

- ФОТ найма по строкам выше (backend + frontend + QA + DevOps + ПМ + UX в части найма): **~9 990 000 ₽** (UX и консультанты часто как подряд — ниже считаем только ядро найма: **9 540 000 ₽**).
- Взносы **30%** от **9 540 000 ₽** ≈ **2 862 000 ₽** (округлённо; для отчёта можно указать диапазон **25–30%** и ссылку на НК РФ).

НДФЛ с сотрудника обычно **не увеличивает** расходы работодателя по зарплате в учётной модели «гросс», но при формировании **полной стоимости человека** для заказчика иногда показывают **гросс + взносы**.

**Итого с взносами (без НДС ОСН отдельной строкой):**

| | ₽ |
|--|--:|
| Прямые затраты подытог | 11 540 000 |
| Страховые взносы (~30% от ФОТ найма 9 540 000) | 2 862 000 |
| **Всего после взносов** | **14 402 000** |

Если часть команды оформлена как **ИП на УСН «доходы» 6%** или **самозанятые**, налоговую нагрузку закладывают **в ставку подрядчика** или отдельными строками **6% / 4–6%** от их счетов — итог может **слегка снизиться или перераспределиться** по статьям.

**НДС:** при общей системе налогообложения исполнителя и заказчателя возможен НДС сверху договора — в учебной смете можно добавить примечание: *«При ОСН сверху возможен НДС 22% к выручке исполнителя по правилам действующего законодательства»* (ставку уточнить на дату отчёта).

### 4.3. Резерв на непредвиденные расходы

Рекомендуется **20%** от суммы **после взносов**:

- 20% × 14 402 000 ₽ ≈ **2 880 400 ₽** → округлим **2 900 000 ₽**

Обоснование: задержки доступа к промышленным БД, доработки безопасности, нестабильность качества NL→SQL, смена приоритетов заказчика.

### 4.4. Итого бюджет проекта (ординир)

| | ₽ |
|--|--:|
| Затраты с учётом страховых взносов | 14 402 000 |
| Непредвиденные (**20%**) | 2 900 000 |
| **ИТОГО (ориентир)** | **~17 300 000** |

Диапазон для формулировки в отчёте: **~15–20 млн ₽** на год разработки полного ТЗ при описанной команде; при сокращении состава или сроков сумма **пропорционально ниже**.
