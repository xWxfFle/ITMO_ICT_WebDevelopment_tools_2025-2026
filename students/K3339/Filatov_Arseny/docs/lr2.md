## Задание 1: сумма чисел от 1 до N

### Идея

Полное перечисление 10 триллионов слагаемых нереализуемо. В каждой программе определена **`calculate_sum(start, end)`** — сумма на отрезке \\([start, end]\\) через формулу:

\\[
\\sum_{i=a}^{b} i = \\frac{b(b+1)}{2} - \\frac{(a-1)a}{2}.
\\]

Интервал `[1; N]` разбивается на несколько последовательных подотрезков; каждый считается в отдельной параллельной задаче. Итог — сумма результатов подзадач.

### Особенности подходов

| Подход | Роль в модели выполнения Python |
|--------|------------------------------------|
| **threading** | Несколько потоков делят общий интерпретатор; из-за **GIL** простые операции над `int` мало параллелятся по CPU |
| **multiprocessing** | Отдельные процессы — отдельные интерпретаторы; настоящий параллелизм на ядрах, но **заметные накладные расходы** на старт процессов и сериализацию аргументов |
| **asyncio** + **`asyncio.to_thread`** | Координация в одном процессе; тяжёлые функции выполняются в **пуле потоков** — по сути близко к threading для числового кода, но демонстрирует идиому async |

### Программы

- `Lr2/task1_sum/sum_threading.py` — `ThreadPoolExecutor`
- `Lr2/task1_sum/sum_multiprocessing.py` — `multiprocessing.Pool.starmap`
- `Lr2/task1_sum/sum_asyncio.py` — `asyncio.gather` + `asyncio.to_thread`

Запуск (рабочая директория — `task1_sum`):

```bash
python sum_threading.py
python sum_multiprocessing.py
python sum_asyncio.py
```

Общее для заданий: **`constants.py`** (N и число чанков), **`math_utils.py`** (деление интервала, эталон `N(N+1)/2`).

### Замеры

| Программа | Время выполнения, с | Параллельных задач |
|-----------|---------------------|---------------------|
| `sum_threading.py` | ~0.0015 | 16 потоков |
| `sum_multiprocessing.py` | ~0.16 | 16 процессов |
| `sum_asyncio.py` | ~0.0035 | 16 вызовов в пуле потоков |

На столь малой по CPU работе доминируют **накладные расходы процессного пула**, поэтому multiprocessing здесь медленнее. Для **долго считающихся** задач без GIL-ограничения выиграл бы уже он. Потоки и `to_thread` сопоставимы порядком.

## Задание 2: парсинг и PostgreSQL ЛР1

### База данных

Подключение: **`DATABASE_URL`** или значение по умолчанию из `practice_1_3` (`db_common.DEFAULT_DB_URL`). Создаётся таблица **`lr2_parsed_web_title`** — отдельно от сущностей Ledger (не ломает схему ЛР1). Поля в частности:

- **`url`** — адрес документа;
- **`title`** — текст из `<title>`;
- **`fetched_at`** — время вставки (UTC).

### Функции и скрипты

В каждом файле реализована **`parse_and_save(url)`**:

| Файл | Сеть | Параллелизм |
|------|------|-------------|
| `parse_threading.py` | **requests** | Несколько потоков по частям списка URL |
| `parse_multiprocessing.py` | **requests** | По одному процессу на часть списка |
| `parse_asyncio.py` | **aiohttp** | Одна `ClientSession` на запуск + `asyncio.gather` по частям |

Спискок URL задаётся в **`urls.py`**. Число частей распределения задаёт переменная окружения **`LR2_PARSE_WORKERS`** (по умолчанию 4). В каждой части запросы к сайтам делаются **последовательно**; параллелизм — между частями (для явного следования условию «разделите список на равные части» при трёх разных технологиях).

### Запуск

```powershell
cd Lr2/task2_web_parse
$env:DATABASE_URL="postgresql://user:password@localhost:5432/filatov_finance_p13"
$env:LR2_PARSE_WORKERS="4"
python parse_threading.py
python parse_multiprocessing.py
python parse_asyncio.py
```

### Замеры

Убедитесь, что экземпляр PostgreSQL из ЛР1 доступен по `DATABASE_URL`. После нескольких прогонов очищайте таблицу или меняйте URL, если нужны «чистые» повторные измерения.

| Программа | Время выполнения, с | Комментарий после прогона |
|-----------|-----------------------|---------------------------|
| `parse_threading.py` | 1.312 | ограниченно эффективен при блокирующих I/O из-за многопоточности под GIL для чистого Python, но `requests`/сеть отпускают GIL |
| `parse_multiprocessing.py` | 0.828 | больше накладных расходов, чем потоков; каждый процесс — свой пул коннектов к БД |
| `parse_asyncio.py` | 2.033 | для сетевого I/O один процесс кооперативно ждёт сокеты; при росте числа параллельных запросов обычно выгоден по памяти и масштабированию относительно процессов |
